**Status**: DRAFT - Defining Intent
**Created**: 2026-02-13
**Related**: UI_REDESIGN_ANALYSIS.md

## Overview

The UI consists of **4 main interfaces**:

| Page | Purpose | Primary User | Data Store |
|------|---------|--------------|------------|
| **Chat UI** | Generate workflow configs conversationally | Developer/Designer | `chat_sessions`, `chat_messages` |
| **Executions** | View and manage workflow execution history | Operator/Developer | `executions`, `execution_states` |
| **Deployments** | Manage deployed workflow containers | Operator | `deployments` |
| **MLflow** | Observability: traces, experiments, metrics | Operator/Developer | MLflow's own DB |

### Features
1. **Capability Introduction**: System explains what workflows are supported
2. **State Machine Visualization**: Show workflow as diagram before generating
3. **Iterative Refinement**: User can modify, ask questions
4. **Enhancement Suggestions**: Tools to use, API keys needed, etc.
5. **Execute Button**: Runtime execution (no persistence)
6. **Deploy Button**: Docker deployment (persisted)
7. **Env Input Area**: Auto-populated with required API keys

### Data Stored

| Table | Purpose |
|-------|---------|
| `chat_sessions` | Conversation sessions with user |
| `chat_messages` | Individual messages in session |

### Schema

```python
# chat_sessions
class ChatSession(Base):
    __tablename__ = "chat_sessions"

    session_id: Mapped[str]       # UUID primary key
    user_identifier: Mapped[str]  # Browser fingerprint
    created_at: Mapped[datetime]
    updated_at: Mapped[datetime]
    generated_config: Mapped[Optional[str]]  # Final YAML
    status: Mapped[str]           # "in_progress", "completed", "abandoned"

# chat_messages
class ChatMessage(Base):
    __tablename__ = "chat_messages"

    id: Mapped[int]
    session_id: Mapped[str]       # FK to chat_sessions
    role: Mapped[str]             # "user" or "assistant"
    content: Mapped[str]
    created_at: Mapped[datetime]
    message_metadata: Mapped[Optional[str]]  # JSON: model, tokens, etc.
```

### Features
1. **Execution List**: All past runs with status, workflow name, duration, cost
2. **Filtering**: By status (completed, failed, running), by workflow name, by date
3. **Execution Detail**: Inputs, outputs, state history, bottleneck analysis
4. **Actions**: Cancel running, Restart failed, View in MLflow
5. **Metrics**: Total tokens, cost, duration per execution
6. **Deployment Link**: If executed on a deployment, link to it

### Data Stored

| Table | Purpose |
|-------|---------|
| `executions` | One record per workflow run |
| `execution_states` | State checkpoints after each node |

### Schema

```python
# executions (was workflow_runs)
class Execution(Base):
    __tablename__ = "executions"

    id: Mapped[str]                    # execution_id
    workflow_name: Mapped[str]
    status: Mapped[str]                # "pending", "running", "completed", "failed"
    config_snapshot: Mapped[Optional[str]]  # YAML used
    inputs: Mapped[Optional[str]]      # JSON inputs
    outputs: Mapped[Optional[str]]     # JSON outputs
    error_message: Mapped[Optional[str]]
    started_at: Mapped[datetime]
    completed_at: Mapped[Optional[datetime]]
    duration_seconds: Mapped[Optional[float]]
    total_tokens: Mapped[Optional[int]]
    total_cost_usd: Mapped[Optional[float]]
    bottleneck_info: Mapped[Optional[str]]
    deployment_id: Mapped[Optional[str]]  # FK if executed on deployment (NEW)

# execution_states
class ExecutionState(Base):
    __tablename__ = "execution_states"

    id: Mapped[int]
    execution_id: Mapped[str]          # FK to executions.id
    node_id: Mapped[str]
    state_data: Mapped[str]            # JSON state
    created_at: Mapped[datetime]
```

### Key Insight
Both runtime and deployed executions go here. The `deployment_id` field indicates if it was run on a deployed container.

### Features
1. **Deployment List**: All registered containers with status, health
2. **Health Monitoring**: Heartbeat-based alive/dead status (HTMX polling every 10s)
3. **Register**: Manually register a running container
4. **Execute**: Run a workflow on a deployment (with env input)
5. **Deregister**: Remove dead/retired deployments
6. **View Schema**: See workflow inputs/outputs schema
7. **View Docs**: Link to container's /docs endpoint

### Schema

```python
# deployments (was agents)
class Deployment(Base):
    __tablename__ = "deployments"

    deployment_id: Mapped[str]         # Unique ID (primary key)
    deployment_name: Mapped[str]       # Human-readable name
    workflow_name: Mapped[str]         # Workflow this deployment runs (NEW)
    host: Mapped[str]                  # Container host
    port: Mapped[int]                  # Container port
    last_heartbeat: Mapped[datetime]   # For TTL-based health
    ttl_seconds: Mapped[int]           # Default: 60
    capabilities: Mapped[Optional[str]] # JSON: tools, features
    deployment_metadata: Mapped[Optional[str]]  # JSON: additional info
    registered_at: Mapped[datetime]

    def is_alive(self) -> bool:
        """Check if heartbeat is within TTL"""
        expiration = self.last_heartbeat + timedelta(seconds=self.ttl_seconds)
        return datetime.utcnow() < expiration
```

## 8. Key Design Decisions

1. **Table names match page names**: `executions`, `deployments` - simple and intuitive
2. **Orchestrator absorbed**: No separate page, all functionality in Deployments
3. **Single app database**: `configurable_agents.db` for all app data
4. **MLflow separate**: `mlflow.db` remains MLflow's own database
5. **Execution â†’ Deployment link**: `deployment_id` FK in executions table (nullable for runtime runs)
6. **Chat UI as entry point**: Primary interface for creating and running workflows
